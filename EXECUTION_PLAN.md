# 当前项目执行计划

## 目标

按既定顺序补齐当前项目的稳定性、移动端体验、工程结构和发布完整度，避免在继续加功能前把技术债继续堆高。

## 执行顺序

1. 补浏览器级 QA / E2E
2. 优化首包和移动端加载
3. 拆分 `LevelScene`
4. 补项目文档
5. 补发布级功能

## 第一阶段：补浏览器级 QA / E2E

### 目标

把当前“逻辑单测通过”的状态升级成“真实页面可自动回归验证”。

### 范围

- 启动页进入游戏
- 第 1 关基础移动
- 键盘跳跃
- 移动端触控按钮输入
- HUD 显示
- 过关场景切换
- 失败后重新开始

### 实施原则

- 先写失败用例，再补最小实现
- 尽量复用现有 Vite 启动方式
- 只增加最小必要的测试观察口
- 不让测试代码污染正式玩法逻辑

### 产出

- 浏览器级冒烟测试脚本
- E2E 运行命令
- 必要的测试辅助接口
- 对应验证记录

### 完成标准

- 本地可一键运行浏览器级回归
- 至少覆盖桌面键盘和移动触控两条路径
- 回归结果可稳定判断通过/失败

## 第二阶段：优化首包和移动端加载

### 目标

降低手机端首开压力，减少弱网和低性能设备上的等待和卡顿。

### 范围

- 分析构建产物体积
- 识别 Phaser 和业务代码的主要体积来源
- 评估是否可拆分场景、数据或初始化逻辑
- 清理不必要的打包内容

### 产出

- 包体分析结论
- 一轮最小有效优化
- 优化前后构建结果对比

### 完成标准

- 主包体积有实质下降
- 不引入玩法回归
- `build`、`test`、`type-check` 保持通过

## 第三阶段：拆分 `LevelScene`

### 目标

降低单文件复杂度，为继续加关卡、机关和 UI 做结构准备。

### 范围

- 玩家更新与碰撞
- 敌人更新与受击逻辑
- 平台交互
- HUD 更新
- 关卡完成/失败流程

### 实施原则

- 只做结构整理，不主动改玩法
- 优先抽离职责边界清晰的逻辑
- 每次拆分后立即跑测试验证

### 产出

- 更小的场景协作模块
- 补充或调整的回归测试

### 完成标准

- `LevelScene` 明显瘦身
- 主要职责有独立模块承接
- 行为保持一致

## 第四阶段：补项目文档

### 目标

让项目可维护、可接手、可继续扩展。

### 范围

- `README.md`
- 项目启动方式
- 控制说明
- 现有功能概览
- 测试命令
- 移动端适配说明
- 已知限制和后续计划

### 产出

- 完整根目录说明文档
- 与当前实现一致的运行说明

### 完成标准

- 新接手的人只看文档就能运行项目
- 文档与代码现状一致

## 第五阶段：补发布级功能

### 目标

把当前可玩 demo 补到更接近正式移动网页游戏的状态。

### 候选功能

- 暂停/继续
- 当前关重开
- 基础设置项
- 音效与背景音乐
- 简单进度保存

### 实施原则

- 先做对体验影响最大的功能
- 每次只引入一类明确能力
- 保持移动端优先

### 完成标准

- 至少补齐一组核心发布能力
- 新能力有对应验证

## 每阶段统一检查项

每完成一个阶段，都执行以下检查：

- `npm run test`
- `npm run type-check`
- `npm run build`
- 同步当前阶段完成情况和剩余问题
